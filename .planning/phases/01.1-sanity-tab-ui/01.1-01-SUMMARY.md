## ModData Shape: Before vs After

**Before (Phase 1):**
```lua
md.SanityTraits = {
    sanity       = <int>,
    appliedStage = "Healthy",
    profession   = <string|"unknown">,
}
```

**After (Plan 01.1-01):**
```lua
md.SanityTraits = {
    sanity        = <int>,
    appliedStage  = "Healthy",
    profession    = <string|"unknown">,
    log           = {},   -- D-11: 50-entry FIFO of sanity events
    appliedTraits = {},   -- D-17: list of traits this mod has applied
}
```

**Idempotent upgrade pattern (rationale for downstream plans):**
Plan 03 panel code can rely on `md.SanityTraits.log` and `md.SanityTraits.appliedTraits` being non-nil arrays after `OnCreatePlayer` has fired. New characters get them seeded; loaded saves get them seeded on the first `OnCreatePlayer` of the new mod version. Subsequent reloads do nothing (the fields are no longer nil). No schema-version counter needed — the additive-only, per-field `== nil` checks make the migration safely idempotent.

## Decisions Made

- **Boundary semantics resolved via lowest-threshold-first ordering.** The `if sanity <= 50 then return "broken"` chain is ordered carefully so a value of 50 returns `"broken"` rather than falling through to `"numb"`. This matches the D-10 contract and was already locked in CONTEXT.md / RESEARCH.md / UI-SPEC.md.
- **Upgrade-path print is one-shot.** A boolean `upgraded` accumulator means the upgrade message fires once per actual migration. Re-loading the same save afterwards stays silent — clean console.txt signal that the migration succeeded without recurring noise.
- **No new files created.** The plan defined inserts/replacements within two existing files. The only new symbols are scoped under `SanityTraits.*` (no global pollution).
