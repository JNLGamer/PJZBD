# Phase 01.1: Sanity Tab UI ‚ÄĒ Context

**Gathered:** 2026-04-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Add a visible **"Psyche"** tab to the in-game character info window (`ISCharacterInfoWindow`) that surfaces the previously-invisible sanity meter built in Phase 1. The tab contains four UI elements:

1. A horizontal **sanity bar** (3-stop red‚Üíyellow‚Üígreen gradient, with always-visible stage threshold tick marks) plus a numeric readout `970 / 1000 (97%)`.
2. A **stage label** computed from the current sanity value using the threshold table this phase locks in (so the label is always correct in v1, and Phase 2 reuses the same `STAGE_THRESHOLDS` for transition logic).
3. An **event log** (last 50 entries, persisted in ModData) recording sanity changes, stage transitions, trait apply/remove, and recovery moments.
4. A **debuff icon row** that reserves space for up to 6 active sanity-applied traits; only filled slots render (no empty placeholder boxes); hover shows full trait info plus "Applied by Sanity Traits at \<stage\>".

**This phase explicitly reverses the v1 "no UI / framework needed first" scope decision** previously documented in PROJECT.md and REQUIREMENTS.md "Out of Scope". After execution, both docs need to move that item from Out-of-Scope ‚Üí Validated.

**Out of scope for this phase (belongs in other phases):**
- Stage transition logic that *applies traits* and *fires events* ‚ÄĒ Phase 2 (uses our `STAGE_THRESHOLDS` and `STAGE_NAMES`)
- Passive sanity decay over time ‚ÄĒ Phase 3
- Occupation-specific threshold modifiers ‚ÄĒ Phase 4
- Habit-based addiction tracking ‚ÄĒ Phase 5
- Sandbox option to disable the tab ‚ÄĒ Phase 6 (deferred)
- Tear-off window support (drag tab into floating window) ‚ÄĒ deferred
- `getText()` i18n / translation ‚ÄĒ deferred
- Sound or visual flair on stage transitions ‚ÄĒ deferred

</domain>

<decisions>
## Implementation Decisions

### Tab integration

- **D-01:** Hook via standard PZ wrap pattern ‚ÄĒ wrap `ISCharacterInfoWindow:createChildren` so vanilla creates its 5 tabs first, then we append ours. This composes cleanly with other mods doing the same wrap (e.g. KillCount).
- **D-02:** Tab label: **"Psyche"** (matches PROJECT.md's "psyche profile" terminology). Code, data, file names, and the mod itself remain "Sanity Traits" ‚ÄĒ only the player-facing tab text is "Psyche."
- **D-03:** Tab position: **end of the strip** (after vanilla's "Clothing Insulation"). Least disruptive to vanilla muscle memory; new tab is discoverable without being intrusive.
- **D-04:** Tab content panel derives from `ISPanelJoypad` and follows the structure of `ISCharacterProtection` (vanilla's Protection tab ‚ÄĒ clean 230-line single-purpose template).

### Bar design

- **D-05:** Sanity bar is a horizontal full-width rectangle with a **3-stop gradient: red ‚Üí yellow ‚Üí green** (red at sanity 0, yellow at midpoint ~500, green at 1000). Use the same `colorScheme` value-stop pattern as `ISCharacterProtection.lua:19-22` for consistency with vanilla.
- **D-06:** Numeric readout next to the bar: format as `"970 / 1000 (97%)"` ‚ÄĒ both raw value and percent, comma-separated.
- **D-07:** Stage threshold ticks **always visible** on the bar ‚ÄĒ small vertical marks at the 4 threshold positions (Sad / Depressed / Traumatized / Desensitized boundaries). Each tick rendered as a 1-2px line in the bar's foreground color.

### Stages (locked in this phase, consumed by Phase 2)

- **D-08:** **`SanityTraits.STAGE_THRESHOLDS`** constant table:
  ```lua
  SanityTraits.STAGE_THRESHOLDS = {
      sad          = 750,  -- ‚Č§750 enters Shaken
      depressed    = 500,  -- ‚Č§500 enters Hollow
      traumatized  = 250,  -- ‚Č§250 enters Numb
      desensitized = 50,   -- ‚Č§50  enters Broken
  }
  ```
  Phase 2 reads this table to fire transitions; this phase reads it to compute the stage label and place threshold ticks.

- **D-09:** **`SanityTraits.STAGE_NAMES`** constant table ‚ÄĒ **thematic names replace the original "Healthy/Sad/Depressed/Traumatized/Desensitized" naming**:
  ```lua
  SanityTraits.STAGE_NAMES = {
      stable  = "Stable",   -- sanity >  750
      shaken  = "Shaken",   -- sanity ‚Č§ 750
      hollow  = "Hollow",   -- sanity ‚Č§ 500
      numb    = "Numb",     -- sanity ‚Č§ 250
      broken  = "Broken",   -- sanity ‚Č§ 50
  }
  ```
  Trait IDs (`base:cowardly`, `base:hemophobic`, etc.) are unchanged ‚ÄĒ only player-facing stage labels are renamed.

- **D-10:** **`SanityTraits.computeStage(sanity)`** helper function returns the stage key (e.g. `"shaken"`) for a given sanity value. Phase 2 reuses this same helper for transition logic. No throwaway code in 01.1.

### Event log

- **D-11:** Event log is stored in `player:getModData().SanityTraits.log` as an array of structured entries. **Persisted via ModData** (survives save/reload).
- **D-12:** Cap at **50 entries**. When a 51st entry is added, drop the oldest (FIFO). Implement as a circular buffer or simple `table.remove(log, 1)` on overflow.
- **D-13:** Log records **all four event categories** the user selected:
  - Sanity changes (every kill in Phase 1; passive decay in Phase 3)
  - Stage transitions (Phase 2 fires these; 01.1 wires the recording hook)
  - Trait applied / removed (Phase 2 fires these; 01.1 wires the recording hook)
  - Recovery moments (sanity going UP ‚ÄĒ Phase 3)
- **D-14:** Each log entry shape:
  ```lua
  { time = <ingame_timestamp>, category = "kill"|"stage"|"trait"|"recovery", delta = <¬ĪN or nil>, text = "Zombie killed", icon = nil|"path" }
  ```
  Display as a scrollable list, newest entries at the top.
- **D-15:** Phase 01.1 wires the **logger API** (`SanityTraits.log(category, text, delta)`). Phase 1's kill handlers get updated to call it. Phase 2/3/5 will call it for their event types ‚ÄĒ no change needed to the logger itself.

### Debuff row

- **D-16:** Debuff row reserves space for **up to 6 trait icons** in a horizontal row at the bottom of the panel. Slot positions are reserved (layout space) but **only slots with an active trait render** ‚ÄĒ no empty placeholder boxes are drawn.
- **D-17:** "Active sanity-applied trait" tracking: `player:getModData().SanityTraits.appliedTraits` array stores `{traitId, appliedAtStage, appliedAtTime}` for each trait this mod has applied. The row renders one icon per entry. Phase 2 populates this; Phase 01.1 ships an empty array and the rendering loop.
- **D-18:** Icon source: vanilla **trait icons** at `ProjectZomboid/media/ui/Traits/`. Each vanilla trait has a corresponding `Trait_<id>.png`. Falls back to a generic placeholder (procedural rect) if a trait icon is missing.
- **D-19:** Hover tooltip on a debuff icon displays:
  - Trait name (vanilla `getText("UI_trait_<id>")`)
  - Trait description (vanilla `getText("UI_trait_<id>_desc")`)
  - "Applied by Sanity Traits at \<stage_name\>" (using the thematic stage names from D-09)

### Refresh, layout, and v1 polish defaults (Claude's discretion)

- **D-20:** Refresh model: panel re-reads sanity in `:render()`, matching the `ISCharacterProtection` pattern. No event subscription needed ‚ÄĒ natural per-frame redraw.
- **D-21:** Internal layout (top-to-bottom): bar + numeric + stage label as a header row ‚Üí event log scrollable list (taking most of the vertical space) ‚Üí debuff row at bottom.
- **D-22:** Player handle uses `getSpecificPlayer(self.playerNum)` (vanilla pattern), not `getPlayer()` ‚ÄĒ even though SP-only, this matches the vanilla code style and keeps a clean SP‚ÜíMP migration path open.
- **D-23:** Strings hardcoded in English in v1; no `getText()` translation hooks. Refactor deferred.

## Existing Code Insights

### Reusable Assets (vanilla)

- **`ISTabPanel:addView(name, view)`** ‚ÄĒ public API on `self.panel` inside `ISCharacterInfoWindow`. Single call to register a new tab.
- **`ISPanelJoypad` base class** ‚ÄĒ derive our SanityPanel from this. Inherits `prerender`, `render`, joypad navigation.
- **`ISCharacterProtection.colorScheme`** pattern (lines 19-22) ‚ÄĒ `{ {val=0, color=red}, {val=100, color=green} }` for value-stop gradients. We use a 3-stop variant for the sanity bar.
- **`bodyPartPanel:setColorScheme()` / `:getRgbForValue()`** ‚ÄĒ vanilla helper that returns interpolated RGB for a value. Could reuse, but a small inline lerp is probably simpler for our case.
- **`ISScrollingListBox`** ‚ÄĒ vanilla scrollable list widget; powers the event log.
- **`ISToolTip`** (or `ISToolTipInv`) ‚ÄĒ vanilla tooltip widget for hover on debuff icons.
- **`getTexture("media/ui/Traits/Trait_<id>.png")`** ‚ÄĒ vanilla trait icons; reuse for debuff row.
- **`getSpecificPlayer(playerNum)`** ‚ÄĒ player handle, vanilla pattern.

### Reusable Assets (this mod, Phase 1)

- **`SanityTraits` global namespace** ‚ÄĒ already exists ([1_SanityTraits_Init.lua](Mods/Sanity_traits/42/media/lua/client/1_SanityTraits_Init.lua)). Constants: `VERSION`, `SANITY_MAX=1000`, `SANITY_MIN=0`, `ZOMBIE_WEIGHT=10`, `SURVIVOR_WEIGHT=30`, `LOG_TAG="[SanityTraits]"`.
- **`SanityTraits.STARTING_SANITY_BY_PROFESSION`** + **`getStartingSanity(profName)`** ‚ÄĒ exists, can be referenced by stage label tooltips later if needed.
- **`player:getModData().SanityTraits = { sanity, appliedStage, profession }`** ‚ÄĒ Phase 1 ModData shape. **This phase extends it to add `log = {}` and `appliedTraits = {}`.** Schema migration: `OnCreatePlayer` handler's idempotent guard already handles new fields gracefully (existing characters get the new fields on next login via a `if md.SanityTraits.log == nil then md.SanityTraits.log = {} end` upgrade pattern).

### Established Patterns

- **Wrap, don't replace, vanilla functions** ‚ÄĒ composability with other mods that do the same.
- **Lua files load alphabetically** ‚ÄĒ numeric prefix (`1_`, `2_`, `3_`, `4_`) controls load order. New UI files should be `4_SanityTraits_UI.lua` or split as `4_SanityTraits_Logger.lua`, `5_SanityTraits_Panel.lua`.
- **B42 mod structure** ‚ÄĒ mod content under `Mods/Sanity_traits/42/media/lua/client/` (already in place; new files go in same directory).
- **`print("[SanityTraits] ...")`** for debug output. Goes to `console.txt` and the in-game Lua console.

### Integration Points

- **`ISCharacterInfoWindow:createChildren`** ‚ÄĒ wrap target. Our `4_SanityTraits_UI.lua` (or similar) installs the wrap at file-load time.
- **`Events.OnZombieDead`** + **`Events.OnWeaponHitXp`** handlers in `3_SanityTraits_KillEvents.lua` ‚ÄĒ already exist; this phase updates them to call `SanityTraits.log()` after decrementing sanity.
- **`Events.OnCreatePlayer`** handler in `2_SanityTraits_ModData.lua` ‚ÄĒ already exists; this phase updates it to seed `log = {}` and `appliedTraits = {}` in the ModData table.
- **No new event subscriptions needed for the panel itself** ‚ÄĒ `:render()` is called per-frame by the parent window; no polling required.

## Specific Ideas

- The user explicitly chose **"Psyche"** as the tab label (matches PROJECT.md's "psyche profile" terminology already used in occupation-archetype design).
