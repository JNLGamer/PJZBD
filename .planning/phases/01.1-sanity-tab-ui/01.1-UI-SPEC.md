# Phase 01.1 — UI Design Contract: "Psyche" Tab

> Visual and interaction contract for the in-game "Psyche" tab added to ISCharacterInfoWindow.
> Generated by gsd-ui-researcher. Consumed by gsd-planner and gsd-executor.
>
> IMPORTANT DOMAIN NOTE: This is a Project Zomboid Build 42 Lua mod UI — not a web app.
> There is no CSS, React, Tailwind, shadcn, or HTML. "Design tokens" map to PZ drawing primitives:
> colors are {r,g,b,a} tables, typography is UIFont.* constants, spacing is integer pixel offsets,
> and "components" are Lua subclasses of ISPanelJoypad / ISScrollingListBox / ISImage.
>
> **AMENDED 2026-04-28** via VERIFICATION GAP-01/GAP-02 closure (Plan 01.1-04).
> The original spec (drafted 2026-04-28 from CONTEXT decisions D-01..D-23) described
> a horizontal full-width bar with single-color fill computed from sanity. After
> in-game UAT, the user requested:
> - GAP-01: bar should be vertical, pinned to the right edge.
>   _User: "I kinda dislike the horisonal bar, i think it should be vertical on the right side"_
> - GAP-02: bar should always render the full green->yellow->red gradient; sanity
>   controls fill level (track-mask), not fill color.
>   _User: "the bar is missing the gradient from green to red. ... it shouldn't slowly turn redder when lower"_
>
> Sections affected: "Color → Gradient interpolation rule (D-05)", "Component Inventory →
> Sanity Bar / Numeric Readout / Stage Label / Event Log", "Layout Contract".
> CONTEXT.md decisions D-05/D-07/D-21 are NOT mutated (history-of-decisions integrity);
> the UI-SPEC visual contract supersedes them for execution.

**Gradient interpolation rule (D-05, AMENDED via GAP-02 closure):**

The full bar is ALWAYS rendered with the complete green->yellow->red gradient
regardless of sanity. Sanity controls fill LEVEL (via track-mask), not fill COLOR.

Render passes (in order):
1. **Gradient backdrop (always-on, never reshapes):** 10 horizontal bands fill the
   full bar height. Band centroid `bandPct = (band - 0.5) / 10`, where `bandPct = 0`
   is the TOP (high sanity = green) and `bandPct = 1` is the BOTTOM (low sanity = red).
   - bandPct < 0.5: r = bandPct/0.5, g = 1, b = 0  (green lerps to yellow)
   - bandPct >= 0.5: r = 1, g = 1 - (bandPct-0.5)/0.5, b = 0  (yellow lerps to red)
2. **Track mask (drawn over empty top portion):** `emptyH = floor(h * (1 - pct))`.
   The dark grey BAR_TRACK rect is drawn from `(x, y)` with width `w` and height `emptyH`.
   This hides the gradient where sanity is "drained." The remaining bottom portion
   (`pct * h` tall) shows the gradient — this IS the visible fill.
3. **Threshold ticks (over track):** 4 horizontal 1px lines at `tickY = y + h - floor(h * threshold/sanityMax)`.
4. **Border (drawn last):** `drawRectBorder` over the full bar area.

Result:
- sanity=1000 (pct=1.0): full gradient visible (green at top, red at bottom).
- sanity=500 (pct=0.5): top half is dark track; bottom half shows yellow->red.
- sanity=0 (pct=0): only dark track + ticks + border visible.

Fill semantics: bottom-up, anchored at the red (low-sanity) end. As sanity drops, the
green portion drains away first; the red portion is the LAST to disappear. This matches
the user's reference image (smooth horizontal green->red gauge) rotated 90°, plus
thermometer/gauge intuition where "danger zone" stays dominant as the meter drains.

Accent reserved for: the bar fill gradient ONLY. No other elements use the red/yellow/green palette. All other UI elements use the monochrome dark grey / off-white palette above.

Destructive color: N/A — no destructive actions exist in this panel.

## Constants Contract

These constants must be added to 1_SanityTraits_Init.lua (loaded first, available to all other files):

```lua
SanityTraits.STAGE_THRESHOLDS = {
    sad          = 750,   -- <=750 enters Shaken
    depressed    = 500,   -- <=500 enters Hollow
    traumatized  = 250,   -- <=250 enters Numb
    desensitized = 50,    -- <=50  enters Broken
}

SanityTraits.STAGE_NAMES = {
    stable  = "Stable",
    shaken  = "Shaken",
    hollow  = "Hollow",
    numb    = "Numb",
    broken  = "Broken",
}

SanityTraits.LOG_MAX_ENTRIES  = 50
SanityTraits.DEBUFF_SLOT_COUNT = 6
```

computeStage boundary semantics (D-10): strictly ordered comparisons, desensitized first:
- sanity <= 50  → "broken"
- sanity <= 250 → "numb"
- sanity <= 500 → "hollow"
- sanity <= 750 → "shaken"
- otherwise    → "stable"

Boundary verification (from VALIDATION.md): computeStage(751) = "stable", computeStage(750) = "shaken", computeStage(51) = "numb", computeStage(50) = "broken".
