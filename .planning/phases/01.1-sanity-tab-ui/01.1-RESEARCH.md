## Summary

Adding a 6th tab to `ISCharacterInfoWindow` is a well-trodden, single-file modding pattern: wrap `:createChildren` so vanilla finishes constructing the 5 vanilla tabs, then call `self.panel:addView("Psyche", sanityPanel)` after `ISCollapsableWindow.createChildren` completes. The vanilla widget API surface is small and stable: `ISTabPanel:addView(name, view)` is a single method call that auto-computes `tabWidth` from the label string, so the only width-handling concern is whether the parent window is wide enough — `ISTabPanel` itself handles tab overflow with built-in scroll arrows.

The `SanityPanel` itself derives from `ISPanelJoypad` (per D-04) and follows the `ISCharacterProtection` template: `:initialise()` → `ISPanelJoypad.initialise + self:create()`, `:createChildren()` builds children, `:render()` redraws every frame. The bar, numeric readout, stage label, and threshold ticks are all procedural draws inside `:render()` (`drawRect`, `drawText`). The event log uses the standard `ISScrollingListBox` widget (initialise → instantiate → addChild → addItem). The debuff row is a horizontal layout of `ISImage` widgets with `setMouseOverText(...)` for hover tooltips — no manual `ISToolTip` instantiation needed.

**Critical discovery (overrides D-18):** vanilla does NOT ship `Trait_<id>.png` files in `media/ui/Traits/` for our negative traits (cowardly, hemophobic, slowhealer, etc.). Only 16 trait PNGs exist there, all profession-related. The vanilla character screen instead obtains the icon via `trait:getTexture()` on a `TraitFactory.Trait` Java object — that texture object is the authoritative source. Plan and executor must use `TraitFactory.getTrait(id):getTexture()` rather than `getTexture("media/ui/Traits/Trait_" .. id .. ".png")`. Procedural fallback is still needed for traits whose Java object returns `nil`.

**Primary recommendation:** Single new Lua file `4_SanityTraits_Panel.lua` containing the logger API + `SanityTraits.STAGE_*` constants + `computeStage()` + the wrap of `ISCharacterInfoWindow:createChildren` + the `SanityPanel = ISPanelJoypad:derive(...)` definition. Modify `2_SanityTraits_ModData.lua` to seed `log = {}` and `appliedTraits = {}`. Modify `3_SanityTraits_KillEvents.lua` to call `SanityTraits.log("kill", "Zombie killed", -10)` after each decrement.

## Architecture Pattern

### The wrap pattern (D-01)

The composability rule is: never replace vanilla functions; wrap them by saving the original to a local and calling it from the new replacement.

```lua
-- in 4_SanityTraits_Panel.lua
local origCreateChildren = ISCharacterInfoWindow.createChildren
function ISCharacterInfoWindow:createChildren()
    origCreateChildren(self)  -- vanilla creates Info, Skills, Health, Protection, ClothingIns
    -- now self.panel exists and has 5 views
    local th = self:titleBarHeight()
    local rh = self:resizeWidgetHeight()
    self.sanityView = SanityPanel:new(0, 8, self.charScreen.width, self.height - 8, self.playerNum)
    self.sanityView:initialise()
    self.panel:addView("Psyche", self.sanityView)
end
```

`addView` (read at `ISTabPanel.lua:457-479`) auto-measures the new tab's width from the label string and pushes it onto `self.viewList`. The 6th tab appears at the end of the strip (D-03 satisfied automatically — append, not insert).

### Panel derivation (D-04)

```lua
SanityPanel = ISPanelJoypad:derive("SanityPanel")

function SanityPanel:initialise()
    ISPanelJoypad.initialise(self)
    self:create()  -- mirrors ISCharacterProtection:initialise (line 9-12)
end

function SanityPanel:createChildren()
    ISPanelJoypad.createChildren(self)
    -- build the event log ISScrollingListBox here
    -- build the 6 ISImage widgets for the debuff row here
end

function SanityPanel:prerender()
    ISPanelJoypad.prerender(self)  -- inherited background draw
end

function SanityPanel:render()
    -- per-frame: re-read sanity, draw bar, ticks, numeric readout, stage label
    -- (event log and debuff icons render themselves as children — already added)
end

function SanityPanel:new(x, y, width, height, playerNum)
    local o = ISPanelJoypad:new(x, y, width, height)
    setmetatable(o, self); self.__index = self
    o.playerNum = playerNum
    o.char = getSpecificPlayer(playerNum)  -- D-22
    o.backgroundColor = {r=0, g=0, b=0, a=0.8}
    o.borderColor = {r=0.4, g=0.4, b=0.4, a=1}
    return o
end
```

This mirrors `ISCharacterProtection.lua:215-232` exactly. The lifecycle is `:new` → `:initialise` (which calls `:create`) → `:createChildren` (called when the panel is added as a child via `panel:addView` → `self:addChild(view)` at `ISTabPanel.lua:466`) → `:prerender`/`:render` per frame.

## Vanilla Widget Reference

### `ISTabPanel` — `ProjectZomboid/media/lua/client/ISUI/ISTabPanel.lua`

We do **not** instantiate this — we call methods on the existing `self.panel` instance inside `ISCharacterInfoWindow`.

| Method | Signature | What it does |
|--------|-----------|--------------|
| `addView(name, view)` | `(string, ISUIElement) → void` | Appends a tab. Auto-measures `tabWidth = MeasureStringX(UIFont.Small, name) + tabPadX`. First view becomes active. ([line 457](../../../ProjectZomboid/media/lua/client/ISUI/ISTabPanel.lua#L457)) |
| `removeView(view)` | `(ISUIElement) → void` | Removes by view reference (not name). ([line 481](../../../ProjectZomboid/media/lua/client/ISUI/ISTabPanel.lua#L481)) |
| `getView(name)` | `(string) → ISUIElement\|nil` | ([line 401](../../../ProjectZomboid/media/lua/client/ISUI/ISTabPanel.lua#L401)) |
| `activateView(name)` | `(string) → bool` | Show this tab, hide others. ([line 411](../../../ProjectZomboid/media/lua/client/ISUI/ISTabPanel.lua#L411)) |
| `getActiveView()` | `() → ISUIElement\|nil` | ([line 438](../../../ProjectZomboid/media/lua/client/ISUI/ISTabPanel.lua#L438)) |

**Key behavior:** `addView` calls `self:addChild(view)` at line 466 and sets `view.parent = self`. The view's `:createChildren` and `:initialise` lifecycle is your responsibility — call `view:initialise()` *before* `panel:addView` (the vanilla `ISCharacterInfoWindow:createChildren` does this for every tab; see `ISCharacterInfoWindow.lua:128, 132, 137, 142, 147`).

**Tab overflow:** `ISTabPanel:render` (lines 105-107) sets a stencil rect when `widthOfAllTabs > self.width` and draws ArrowLeft/ArrowRight scroll buttons. **Adding a 6th tab does NOT require any width adjustment in our code** — the parent window's `tabTotalWidth` calculation in `ISCharacterInfoWindow:createChildren` (lines 107-113) is just the *initial* width hint to ISTabPanel; if our 6th tab makes the bar overflow, the user just gets scroll arrows. The `tabPadX` default is 20 (line 615), so the "Psyche" label adds ~`MeasureStringX("Psyche", UIFont.Small) + 20` ≈ 60px.

**Recommendation on tab width:** Don't try to bump `tabTotalWidth` from inside our wrap — it's a local variable inside vanilla `:createChildren` and isn't accessible after the call. If overflow becomes a real visual problem in UAT, we can extend the parent window's width *after* the wrap by calling `self:setWidth(self.width + extra)`, but vanilla's scroll arrows are the cheaper-by-design fallback.

### `ISToolTip` — `ProjectZomboid/media/lua/client/ISUI/ISToolTip.lua`

We **do not need to instantiate this directly** for v1. Two simpler vanilla paths:

1. **`ISImage:setMouseOverText(text)`** ([ISImage.lua:13](../../../ProjectZomboid/media/lua/client/ISUI/ISImage.lua#L13)) — single-line string tooltip, fires automatically when the engine detects mouse-over. tip:setVisible(false)
tip:setAlwaysOnTop(true)

## Data Model

### `1_SanityTraits_Init.lua` additions

```lua
-- Stage threshold table (LOCKED in 01.1, consumed by Phase 2 transition logic and 01.1 tick rendering)
SanityTraits.STAGE_THRESHOLDS = {
    sad          = 750,  -- ≤750 enters Shaken
    depressed    = 500,  -- ≤500 enters Hollow
    traumatized  = 250,  -- ≤250 enters Numb
    desensitized = 50,   -- ≤50  enters Broken
}

-- Player-facing thematic names (rename of the conceptual stages)
-- Trait IDs are unchanged; this is UI strings only
SanityTraits.STAGE_NAMES = {
    stable  = "Stable",   -- sanity > 750
    shaken  = "Shaken",   -- sanity ≤ 750
    hollow  = "Hollow",   -- sanity ≤ 500
    numb    = "Numb",     -- sanity ≤ 250
    broken  = "Broken",   -- sanity ≤ 50
}

-- Returns the stage key (e.g. "shaken") for a given sanity value.
-- Used by 01.1 panel to render the stage label and by Phase 2 transition logic.
function SanityTraits.computeStage(sanity)
    if sanity <= SanityTraits.STAGE_THRESHOLDS.desensitized then return "broken" end
    if sanity <= SanityTraits.STAGE_THRESHOLDS.traumatized  then return "numb" end
    if sanity <= SanityTraits.STAGE_THRESHOLDS.depressed    then return "hollow" end
    if sanity <= SanityTraits.STAGE_THRESHOLDS.sad          then return "shaken" end
    return "stable"
end

-- Maximum entries kept in the per-player event log (FIFO, oldest dropped on overflow)
SanityTraits.LOG_MAX_ENTRIES = 50

-- Maximum simultaneously-rendered debuff icons in the row
SanityTraits.DEBUFF_SLOT_COUNT = 6
```

### Log entry shape (D-14)

```lua
{
    time     = getTimestamp(),       -- in-game UNIX-style ms; or use getGameTime():getWorldAgeHours() for relative
    category = "kill"|"stage"|"trait"|"recovery",
    delta    = -10 or +5 or nil,     -- the sanity change, sign included; nil for stage/trait events with no numeric
    text     = "Zombie killed",      -- pre-formatted display string
    icon     = nil or "media/...",   -- optional texture path; for v1 we leave nil and render text-only

```

**Recommendation on `time`:** use `getTimestampMs()` (vanilla returns ms since epoch; deterministic across saves). Display formatting (e.g. "Day 3 • 14:30" via `getGameTime():getDay()` and `getHour()`) is Claude's discretion per D-21 / Deferred — for v1, format as `"D" .. day .. " " .. string.format("%02d:%02d", hour, minute)` at render time, computed from `getGameTime()` (not from the entry's `time` — that's just for sort/dedup integrity).

### Logger API (D-15)

```lua
function SanityTraits.log(category, text, delta)
    -- assert/guard the category to one of the four allowed strings
    local validCategories = { kill=true, stage=true, trait=true, recovery=true }
    if not validCategories[category] then
        print(SanityTraits.LOG_TAG .. " WARN: invalid log category '" .. tostring(category) .. "'")
        return
    end
    local player = getPlayer()
    if not player then return end
    local md = player:getModData()
    if not md.SanityTraits or not md.SanityTraits.log then return end

    local entry = {
        time     = getTimestampMs(),
        category = category,
        delta    = delta,        -- may be nil for non-sanity-change categories
        text     = text,
        icon     = nil,          -- v1 hardcodes nil; later phases can pass texture paths
    }
    -- newest at top: insert at position 1
    table.insert(md.SanityTraits.log, 1, entry)
    -- FIFO cap
    while #md.SanityTraits.log > SanityTraits.LOG_MAX_ENTRIES do
        table.remove(md.SanityTraits.log)  -- removes the last element (oldest)
    end

    print(SanityTraits.LOG_TAG .. " log[" .. category .. "] " .. text
        .. (delta and (" (Δ" .. tostring(delta) .. ")") or ""))
end
```

## Refresh Strategy (D-20)

The panel re-reads sanity in `:render()`. The event log listbox is the trickier one — we don't want to repopulate the entire listbox every frame.

**Recommendation:** track a `self.lastLogCount` on the panel. In `:prerender` (or at top of `:render`):

```lua
local md = self.char:getModData().SanityTraits
if md and md.log and #md.log ~= self.lastLogCount then
    -- log changed since last frame — rebuild the listbox
    self.eventLog:clear()
    for i, entry in ipairs(md.log) do
        local label = self:formatLogEntry(entry)  -- "D3 14:30 • Zombie killed (-10)"
        self.eventLog:addItem(label, entry, nil)
    end
    self.lastLogCount = #md.log
end
```

This keeps per-frame work to a single integer comparison in the steady state. The `appliedTraits` row gets the same treatment (compare `#md.appliedTraits` against `self.lastAppliedCount`).

## Tab-Strip Width Handling (D-03 / Open Question from CONTEXT)

The CONTEXT mentioned bumping `tabTotalWidth`. After reading `ISTabPanel.lua` end-to-end:

- `tabTotalWidth` in `ISCharacterInfoWindow:createChildren` (lines 107-113) is just the **initial size hint** to the `ISTabPanel:new(...)` call. It's a local variable; once the constructor runs, the value is locked in `self.panel.width`.
- `ISTabPanel` already handles overflow gracefully via `:render` lines 105-107 (stencil rect) and lines 169-182 (ArrowLeft/ArrowRight scroll buttons).
- So **no width bump is required from our wrap**. If the user complains visually, we can `self:setWidth(self.width + extra)` *after* `origCreateChildren(self)` returns (where `extra ≈ MeasureStringX(UIFont.Small, "Psyche") + UI_BORDER_SPACING + 6`) and also `self.panel:setWidth(self.panel.width + extra)`. But default behavior is fine.

**Ship default behavior; revisit only if UAT shows visual regression.**

## Common Pitfalls

### Pitfall 1: Trait icon paths don't exist

**What goes wrong:** Following D-18 literally and writing `getTexture("media/ui/Traits/Trait_" .. traitId .. ".png")` returns `nil` for every Sanity-applied trait (cowardly, hemophobic, slowhealer, weakstomach, insomniac, disorganized, etc.) because vanilla doesn't ship those PNGs. Only profession-related icons exist in `media/ui/Traits/` (16 files total).

**Why it happens:** The vanilla character creation screen renders trait icons via `trait:getTexture()` on a Java-side `TraitFactory.Trait` object — the texture is embedded in the trait registration, not always loaded from a PNG path. CONTEXT.md author saw the directory exists and assumed the naming convention `Trait_<id>.png` matched. It doesn't.

**How to avoid:** Use `TraitFactory.getTrait(traitId):getTexture()` (the same call vanilla uses at `ISCharacterScreen.lua:606, 137`). Add a nil guard:

```lua
local function getTraitTexture(traitId)
    local trait = TraitFactory.getTrait(traitId)
    if trait then return trait:getTexture() end
    return nil
end

-- in render:
local tex = getTraitTexture(entry.traitId)
if tex then
    self:drawTexture(tex, x, y, 1, 1, 1, 1)
else
    -- procedural fallback per D-18: small filled rect with first letter
    self:drawRect(x, y, 18, 18, 1, 0.3, 0.3, 0.3)
    self:drawRectBorder(x, y, 18, 18, 1, 0.6, 0.6, 0.6)
    self:drawText(string.sub(entry.traitId, -1), x+6, y+1, 1, 1, 1, 1, UIFont.Small)
end
```

**Warning signs:** debuff row shows up empty even after Phase 2 applies traits; console shows `getTexture(...) returned nil` if you log it.

### Pitfall 2: Wrap captures function before it exists

**What goes wrong:** If `4_SanityTraits_Panel.lua` somehow loads before `ProjectZomboid/media/lua/client/XpSystem/ISUI/ISCharacterInfoWindow.lua`, the line `local origCreateChildren = ISCharacterInfoWindow.createChildren` captures `nil`, then later `origCreateChildren(self)` errors.

**Why it happens:** Mod load order vs vanilla load order. In B42, vanilla `lua/client/` files are loaded *before* mod `client/` files (verified by mod loading docs and existing Phase 1 behavior — Phase 1's wrap of `Events.OnZombieDead.Add` works the same way and never fails).

**How to avoid:** Defensive guard:

```lua
if not ISCharacterInfoWindow then
    print(SanityTraits.LOG_TAG .. " ERROR: ISCharacterInfoWindow not loaded; cannot install Psyche tab")
    return
end
local origCreateChildren = ISCharacterInfoWindow.createChildren
```

**Warning signs:** console shows `attempt to index a nil value` at mod load.

### Pitfall 3: ISScrollingListBox `removeFirst` is broken

**What goes wrong:** `ISScrollingListBox:removeFirst` at line 233 calls `table.remove(self.items, 0)` — **0 is not a valid Lua index** (Lua tables are 1-indexed). The call is a no-op; the list grows unbounded.

**How to avoid:** Use `:removeItemByIndex(1)` (line 215, correctly implemented) or do the FIFO bookkeeping in our `SanityTraits.log()` function on `md.SanityTraits.log` directly (recommended — keeps the list source-of-truth in ModData, listbox is a derived view).

**Warning signs:** the listbox keeps growing past 50 items even though logger looks correct.

### Pitfall 4: `:instantiate()` skipped on ISScrollingListBox

**What goes wrong:** `ISScrollingListBox:initialise()` (line 13) is just a passthrough to `ISPanelJoypad.initialise`. The Java-side UIElement creation and scrollbar setup happens in `:instantiate()` (line 51-62). If you skip it, the listbox renders without a scrollbar and may crash on `getScrollHeight()`.

**How to avoid:** always call `:initialise()` then `:instantiate()` then `:addChild(...)`. The order matters.

### Pitfall 5: Re-populating the listbox every frame

**What goes wrong:** Naive `:render` clears and re-adds all 50 entries each frame. Smooth scrolling resets every frame; selection state is lost; CPU wastes time on string measurement.

**How to avoid:** the `lastLogCount` cache pattern in *Refresh Strategy* above. Only rebuild when `#md.log` changed.

### Pitfall 6: Panel constructor calls `getSpecificPlayer` too early

**What goes wrong:** `ISCharacterInfoWindow:createChildren` runs during the player-info-window construction flow. `getSpecificPlayer(0)` should return a valid player by then (vanilla uses it the same way at line 127 for `ISCharacterScreen:new`), but in some race conditions on first character spawn it can return `nil`.

**How to avoid:** mirror vanilla exactly — call `getSpecificPlayer(self.playerNum)` inside `:new`, and guard `:render` against `self.char == nil` (return early, optionally re-attempt `getSpecificPlayer` next frame).

### Sanity bar with 3-stop gradient + threshold ticks

```lua
function SanityPanel:drawSanityBar(x, y, w, h, sanity, sanityMax)
    local pct = sanity / sanityMax  -- 0..1

    -- Background (dark, full width)
    self:drawRect(x, y, w, h, 1, 0.1, 0.1, 0.1)

    -- Foreground gradient: red (0) → yellow (0.5) → green (1)
    local r, g, b
    if pct < 0.5 then
        local t = pct / 0.5
        r, g, b = 1.0, t, 0.0
    else
        local t = (pct - 0.5) / 0.5
        r, g, b = 1.0 - t, 1.0, 0.0
    end
    self:drawRect(x, y, w * pct, h, 1, r, g, b)

    -- Border
    self:drawRectBorder(x, y, w, h, 1, 0.4, 0.4, 0.4)

    -- D-07 threshold ticks (always visible)
    local thresholds = SanityTraits.STAGE_THRESHOLDS
    for _, threshold in pairs({thresholds.sad, thresholds.depressed,
                               thresholds.traumatized, thresholds.desensitized}) do
        local tx = x + math.floor(w * (threshold / sanityMax))
        self:drawRect(tx, y, 1, h, 1, 0.9, 0.9, 0.9)  -- 1px white tick
    end
end

-- in :render():
local md = self.char:getModData().SanityTraits
if not md then return end
local sanity = md.sanity
local barX, barY, barW, barH = 10, 10, self.width - 20, 14
self:drawSanityBar(barX, barY, barW, barH, sanity, SanityTraits.SANITY_MAX)

-- D-06 numeric readout
local pct = math.floor((sanity / SanityTraits.SANITY_MAX) * 100)
local readout = string.format("%d / %d (%d%%)", sanity, SanityTraits.SANITY_MAX, pct)
self:drawText(readout, barX + barW + 6, barY, 1, 1, 1, 1, UIFont.Small)

-- Stage label
local stageKey = SanityTraits.computeStage(sanity)
local stageName = SanityTraits.STAGE_NAMES[stageKey]
self:drawText("Stage: " .. stageName, barX, barY + barH + 4, 1, 1, 1, 1, UIFont.Medium)
```

### Debuff row with tooltip

```lua
function SanityPanel:createChildren()
    ISPanelJoypad.createChildren(self)

    -- Pre-create 6 ISImage slots; show/hide based on appliedTraits length
    self.debuffSlots = {}
    local slotSize = 24
    local slotGap = 4
    local rowY = self.height - slotSize - 10
    for i = 1, SanityTraits.DEBUFF_SLOT_COUNT do
        local img = ISImage:new((i-1) * (slotSize + slotGap) + 10, rowY, slotSize, slotSize, nil)
        img:initialise()
        img:instantiate()
        img:setVisible(false)  -- D-16: only filled slots render
        self:addChild(img)
        self.debuffSlots[i] = img
    end
end

-- in :prerender or :render:
function SanityPanel:refreshDebuffRow()
    local md = self.char:getModData().SanityTraits
    if not md or not md.appliedTraits then return end
    if #md.appliedTraits == self.lastAppliedCount then return end

    for i, slot in ipairs(self.debuffSlots) do
        local entry = md.appliedTraits[i]
        if entry then
            local trait = TraitFactory.getTrait(entry.traitId)
            if trait then
                slot.texture = trait:getTexture()
                local stageName = SanityTraits.STAGE_NAMES[entry.appliedAtStage] or entry.appliedAtStage
                slot:setMouseOverText(
                    trait:getLabel() .. "\n" ..
                    trait:getDescription() .. "\n\n" ..
                    "Applied by Sanity Traits at " .. stageName
                )
                slot:setVisible(true)
            else
                slot:setVisible(false)
            end
        else
            slot:setVisible(false)
        end
    end
    self.lastAppliedCount = #md.appliedTraits
end
```

## Risks and Gotchas

### Risk 1: D-18 trait icon assumption is wrong (HIGH impact, MEDIUM effort to fix)

Already detailed in *Pitfalls → Pitfall 1*. **Action:** the planner should explicitly call out the `TraitFactory.getTrait(id):getTexture()` correction in the plan file. Don't let the executor follow CONTEXT.md D-18 literally.

### Risk 2: Tab strip overflow at 6 tabs (LOW impact, LOW effort to fix)

Six tabs may overflow the default character window width. ISTabPanel handles this via scroll arrows — visually acceptable but not great. **Action:** ship default; revisit only if UAT users report it. If we need to fix, bump `self:setWidth(self.width + extraPx)` and `self.panel:setWidth(...)` in the wrap.

### Risk 3: Renderer performance from naive listbox repopulation (LOW impact, LOW effort to fix)

Already detailed in *Pitfalls → Pitfall 5*. Use the `lastLogCount` cache pattern.

### Risk 4: `ISScrollingListBox:removeFirst` bug (LOW impact, LOW effort to fix)

Already detailed in *Pitfalls → Pitfall 3*. Use `removeItemByIndex(size())` to evict the oldest, or — better — do FIFO on the `md.log` source array and let the listbox rebuild from it.

### Risk 5: Save/reload of pre-01.1 characters (LOW impact, MITIGATED)

Existing Phase 1 saves have `md.SanityTraits = {sanity, appliedStage, profession}` without `log` or `appliedTraits`. The idempotent upgrade in `2_SanityTraits_ModData.lua` (described above) handles this on next `OnCreatePlayer` fire (B42 fires this for both new chars *and* loaded saves per Phase 1 research). **Action:** include the upgrade in the plan; UAT step "load Phase 1 save → check log/appliedTraits exist" is required.

### Risk 6: TraitFactory not loaded at panel construction (LOW impact, NIL effort)

`TraitFactory` is loaded by vanilla early; no risk in practice. Calls happen at `:render` time, well after init.

### Risk 7: ISTabPanel render quirk — first tab is auto-active (NIL impact, NIL effort)

`addView` sets `self.activeView = viewObject` only when `#self.viewList == 1` (line 471). Since vanilla adds 5 views before our 6th, our tab is **not** active by default — user must click it. This is the desired behavior (don't take focus away from vanilla Info tab on game start).

### Risk 8: Joypad navigation incomplete (LOW impact, n/a for v1)

`ISPanelJoypad` defaults work for vanilla widgets that register themselves with `joypadButtonsY`. We don't register the bar (procedural draw) or the listbox or icon-row, so joypad navigation cycles to the panel and stops. Acceptable for v1 — singleplayer + mouse-first per PROJECT.md constraints.
