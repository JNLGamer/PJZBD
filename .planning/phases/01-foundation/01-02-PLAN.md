    ```lua

    -- OnCreatePlayer handler — initializes per-character SanityTraits ModData (CORE-01, CORE-02)
    -- Fires for BOTH new characters and loaded saves; the `if md.SanityTraits == nil` guard prevents overwrite on reload.
    -- Source: 01-RESEARCH.md Pattern 2; ProjectZomboid/media/lua/client/ISUI/PlayerData/ISPlayerData.lua:203
    local function onCreatePlayer(playerIndex, player)
        if not player then return end
        local md = player:getModData()
        if md.SanityTraits ~= nil then
            -- Already initialized (loaded save). Do nothing — preserve persisted sanity.
            return
        end

        -- Pitfall 2 guard: getCharacterProfession() can be nil very early in character creation
        local profName = nil
        local desc = player:getDescriptor()
        if desc and desc:getCharacterProfession() then
            profName = desc:getCharacterProfession():getName()
        end

        local startSanity = SanityTraits.getStartingSanity(profName)

        md.SanityTraits = {
            sanity       = startSanity,
            appliedStage = "Healthy",
            profession   = profName or "unknown",
        }

        print(SanityTraits.LOG_TAG .. " OnCreatePlayer: profession=" .. tostring(profName)
            .. " startingSanity=" .. tostring(startSanity))
    end

    Events.OnCreatePlayer.Add(onCreatePlayer)
    ```

    Critical constraints (all from 01-RESEARCH.md):
    - The handler signature is `function(playerIndex, player)` — both params provided by `Events.OnCreatePlayer`. Do NOT call `getPlayer()` here; the handler receives the player directly. (Source: ISPlayerData.lua:203.)
    - The `if md.SanityTraits ~= nil then return end` guard is MANDATORY. Without it, loading a save would reset the player's sanity. RESEARCH.md anti-pattern: "Initializing ModData in OnNewGame only" — we use OnCreatePlayer + the nil-guard precisely so the same code path serves new and loaded games.
    - The descriptor nil-check is MANDATORY (Pitfall 2): `if desc and desc:getCharacterProfession() then ... end`. Without it, the handler can throw `attempt to index a nil value (field 'getCharacterProfession')` during character creation.
    - The ModData shape MUST be: `{ sanity = <number>, appliedStage = "Healthy", profession = <string> }`. Phase 2 mutates `appliedStage`; Phase 4 reads `profession`. Do not omit any of these three fields.
    - Use `Events.OnCreatePlayer.Add(...)` (NOT `OnNewGame`, NOT `OnGameStart` — see RESEARCH.md "Standard Stack" rationale).
    - Log line MUST start with `SanityTraits.LOG_TAG ..` (the bracketed `[SanityTraits]` tag from 01-01) so console grep is consistent across the mod.
    - Do NOT clamp sanity here — clamping is the kill-event handlers' responsibility (Plan 01-03).
  </action>
